# 账号检测：额度阈值关闭与全部开关

## 功能说明

在「账号检测」页新增账号状态开关能力，复用现有「当前检测分组」范围逻辑：

1. **5小时 / 7天额度阈值输入**：检测达阈值时按百分比判定。
2. **检测达阈值账号**：按当前阈值扫描「当前检测分组」内账号。
3. **关闭达阈值账号**：将最近一次「检测达阈值账号 / 完整检测」命中的账号批量设为 `inactive`。
4. **全部关闭 / 全部开启**：对「当前检测分组」范围内全部账号批量设为 `inactive` / `active`。

## 使用流程

1. 填写网站地址与管理员 API Key，同步账号和分组。
2. 选择 **当前检测分组**（与检测共用同一下拉框；选「不限」表示全部已同步账号）。
3. 设置 **5小时阈值(%)**、**7天阈值(%)**（默认 99）。
4. 点击 **检测达阈值账号** 或 **完整检测**。
5. 可按需：
   - **关闭达阈值账号**：关闭检测结果中的达阈值账号。
   - **全部关闭**：关闭当前检测分组内全部账号。
   - **全部开启**：开启当前检测分组内全部账号。

所有写操作均有二次确认弹窗，执行后会同步刷新本地账号缓存。

## 实现要点

| 项 | 说明 |
|----|------|
| 范围复用 | `iter_detection_target_accounts()` / `delete_group_var`，与检测、显示列表同一分组逻辑 |
| 状态 API | `POST /admin/accounts/bulk-update`，请求体含 `account_ids` + `status` + `schedulable` |
| 通用方法 | `bulk_set_accounts_status(...)`；原 `bulk_mark_accounts_error` 改为调用它并固定 `status=error` |
| 达阈值来源 | `invalid_quota_account_ids_cache`（与「显示达阈值账号 / 直接删除达阈值」同源） |
| 关闭/开启含义 | 关闭 = `status=inactive` **且** `schedulable=false`；开启 = `status=active` **且** `schedulable=true` |

### 已知问题与修复（关闭不生效）

根因：sub2api 管理端写入的停用状态是 `inactive`，但调度快照即时同步条件里写的是 `disabled`/`error`，或 `schedulable=false`。  
只改 `status=inactive` 时，数据库状态会变，但 **schedulable 仍为 true**，管理后台“参与调度”开关看起来仍开启，调度缓存也可能未及时刷新，表现为“点了关闭实际还能调度”。

修复：关闭/全部关闭时同时提交 `schedulable=false`；开启/全部开启时同时提交 `schedulable=true`。

## 相关接口

### 批量更新账号状态

- **方法 / 路径**：`POST /admin/accounts/bulk-update`
- **请求示例**：

```json
{
  "account_ids": [101, 102, 103],
  "status": "inactive",
  "schedulable": false
}
```

`status` 取值：`active` | `inactive` | `error`。  
关闭务必带 `schedulable: false`；开启带 `status: active` 与 `schedulable: true`。

- **成功时客户端使用的字段**（兼容多种返回形态）：
  - `success_ids`：成功更新的账号 ID 列表（若无则按本批请求 ID 计为成功）
  - `results[]`：含 `account_id` / `success` / `error` 的明细（可选）

- **工具侧汇总返回给界面日志的结构**（本地组装，非服务端原样）：

```json
{
  "updated_ids": [101, 102],
  "failed": [{"account_id": 103, "error": "..."}],
  "updated_count": 2,
  "failed_count": 1,
  "status": "inactive",
  "requested_count": 3,
  "scope": "当前检测分组：全部账号（不限）",
  "synced_account_count": 120
}
```

### 用量检测（已有，关闭达阈值依赖其结果）

- **方法 / 路径**：`GET /admin/accounts/{id}/usage`（可带 `?source=passive|active`）
- 客户端用 5 小时 / 7 天阈值判断是否写入 `invalid_quota_account_ids_cache`。

## 界面布局（账号检测页）

按流程分区，一眼能看懂：

1. **① 准备**：选分组、同步、阈值%、并发  
2. **② 检测**：检测达阈值 / 401/403 / 完整检测（只记结果，不自动删）  
3. **③ 处理结果**：达阈值显示/关闭/删除；401 显示/删除；本分组全部关闭/开启  
4. **④ 账号列表**：结果展示区（并弹窗）；可清空或删除列表中的账号  

关闭达阈值：`inactive` + `schedulable=false`。全部开启：`active` + `schedulable=true`。

## 阈值持久化

`5小时阈值%` / `7天阈值%` 与网站地址一样写入本地 `gui-config.json`：

- 字段：`five_hour_threshold`、`seven_day_threshold`（非负浮点数，默认 `99`）
- 点「保存配置」或关闭窗口时自动保存
- 下次启动自动载入

## 注意事项

- 「全部关闭/开启」依赖已同步的 `accounts_cache`；未同步会提示先同步。
- 「关闭达阈值账号」依赖最近一次无额度类检测结果；未检测会提示先检测。
- 批量状态更新按每批最多 100 个账号提交，支持任务取消。
